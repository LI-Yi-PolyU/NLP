## 5. 结果解读

### 5.1 已验证通过

1. 数据流水线可稳定执行，输入资产完备。
2. 评测与证据导出链路完整可复现。
3. 游戏核心闭环（意图->状态->一致性->生成->结局）可跑通。
4. 一致性自动修复机制在实际轨迹中有证据样例。

### 5.2 主要风险

1. 指标偏乐观风险
- 意图/一致性 1.0 很可能受同分布数据影响，泛化能力未充分验证。

2. 性能风险
- 当前端到端平均时延 18s~27s（两种口径均较高），与“实时”目标有偏差。

3. 生成稳定性风险
- 轨迹中存在非中文叙事（11 条），说明输出约束仍有漏网场景。

4. 对外部模型依赖风险
- 抽测中多轮由 api_fallback 驱动，若网络/API 波动，体验和成本都会受影响。

## 6. 项目设计评估

### 6.1 设计优点（架构层面）

1. 分层清晰：NLU、状态管理、一致性、生成、评测、演示解耦较好。
2. 容错完备：多级回退、规则护栏、SCRIPTED 兜底提高了可演示性。
3. 可解释性强：每轮可追踪意图来源、一致性结果与检索来源。
4. 工程闭环完整：从数据构建到评测再到证据导出形成完整流程。

### 6.2 设计短板（当前版本）

1. 离线评测与线上真实分布之间仍有鸿沟。
2. 在线路径对外部生成模型依赖较高，本地能力占比不足。
3. 端到端性能优化未完成，影响用户交互连续性。
4. 质量约束（如中文输出）主要靠后处理，根因控制还不够强。

### 6.3 综合评价

综合评价：课程项目层面属于“完成度高、架构正确、可答辩”的实现。

建议分档：
- 工程完整度：优秀
- 架构合理性：优秀
- 真实性能表现：中等
- 泛化可信度：中等偏上（需补 OOD/真实集）

参考结论分：13.0~13.8 / 15（取决于答辩对真实性能与泛化的权重）。

## 7. 改进优先级（建议按周执行）

### P0（必须优先）

1. 增加 out-of-template 与人工标注测试集，单独报分。
2. 将 latency 按阶段拆分（NLU/一致性/RAG/生成），定位耗时瓶颈。
3. 增加 API 不可用压测与降级策略指标（成功率、平均响应时间、兜底命中率）。

### P1（一周内）

1. 提升本地 NLU 权重，降低 api_fallback 依赖。
2. 强化中文输出约束（提示词+schema+后处理三重一致）。
3. 建立固定回归集，防止新改动破坏主线可达性。

### P2（后续优化）

1. 引入更细粒度的人类评测协议（可玩性、连贯性、角色一致性）。
2. 增加结构化日志与可观测面板（错误类型、冲突规则命中、回退原因）。
