### **动机：为何选择此项目**

**与课程一致的学习成果**
- **文本数据工作**：收集、清洗、结构化、标注
- **语言生成**：上下文感知生成与可控性
- **交互系统**：意图理解、状态跟踪、实时响应
- **专业交付物**：精致的演示+清晰的技术写作

**为何选择交互式文字冒险设定**
- 自然结合**NLU**（理解）+ **NLG**（生成）+ **记忆/状态**
- 易于评估：分支差异、一致性违规、响应时间
- 比独立文本生成更接近现实世界的"生成式系统工程"

*重点不是"写一个很酷的故事"，而是构建一个**稳健的NLP驱动的交互式叙事系统**。*

### **任务定义（StoryWeaver）**

**具有动态剧情生成的AI驱动文字冒险**
- **输入**：玩家选择/玩家文本输入（例如，"与长者谈判"、"探索洞穴"）
- **系统**：解释输入、维护叙事状态、生成连贯的下一段剧情
- **输出**：下一段剧情（叙述/对话）+ 下一步选项（或自由输入的指导）

**关键特性**
- 动态分支
- 上下文连贯性
- 多轮交互

**主要挑战**
- 多样化输入
- 可控的分支
- 长期一致性

**课堂预期**
- 8分钟演讲
- 实时演示
- 解释方法+有效性

---

### **您应涵盖的核心NLP子任务**

**NLU：意图识别**
- 将玩家输入映射到**动作/意图**类型（例如，NEGOTIATE、EXPLORE、ASK）
- 可选：槽位/实体提取（角色、地点、物品、目标）
- 低置信度处理：澄清、回退或更安全的选项

**NLG：上下文感知生成**
- 生成必须反映先前事件、角色、地点和约束
- 推荐结构化输出：叙述、对话、下一步选择
- 分支必须是可观察的：不同选择→不同后果

**剧情一致性维护**
- 跟踪和保存事实（角色状态、物品、时间线、已揭示信息）
- 核心能力：**检测矛盾** + **修复或拒绝** + **解释**

### **所需开发步骤（使您的报告与这些对齐）**

**1）数据准备**
- 收集与组织：冒险脚本、分支叙事、对话数据、一致性标注
- 预处理：清洗、划分剧情单元、为一致性标注叙事逻辑
- 目标：可复现且可评估的训练/测试

**2）算法设计**
- 上下文感知生成、意图识别、一致性检测、对话管理
- 高效的实时流水线：延迟、缓存、上下文窗口控制
- 分支策略：选择应有意义地改变剧情

**3）系统实现**
- 建议技术栈：Transformers、PyTorch、Gradio
- 界面：输入→生成→下一步选择
- 日志记录：存储轨迹以进行评估和写作

**4）性能评估**
- 叙事质量：连贯性/一致性
- 响应性：响应时间
- 选择匹配准确性+用户满意度

### **评估标准（总分15%；5个标准×3%）**

| 标准     | "顶级"通常的表现                                           | 失分的典型原因                                  |
| -------- | ---------------------------------------------------------- | ----------------------------------------------- |
| 适当性   | 任务设定、挑战、方法和功能与规范良好对齐                   | 看起来像通用聊天/续写；分支薄弱；缺少一致性设计 |
| 严谨性   | 清晰的端到端流程：数据→方法→系统→评估；可复现的设置        | 无评估；设置不清晰；除演示外无证据              |
| 吸引力   | 引人入胜且创新的演示点（例如，受控分支+一致性修复）        | 分支看起来相似；输出单调；无亮点                |
| 演讲表现 | 专业结构、清晰的视觉、稳定的实时演示、良好的节奏           | 幻灯片拥挤；超时；演示不稳定且无应急方案        |
| 写作     | 清晰、简洁、格式合规的报告，解释充分的图表和适当的参考文献 | 格式不合规；解释薄弱；缺少贡献说明              |

### **课堂项目PPT：推荐结构（8-10页幻灯片）**

**目标：在8分钟内解释"是什么/如何做/效果如何/实时运行"**

| #    | 幻灯片                | 必须包含                                     | 建议时间 |
| ---- | --------------------- | -------------------------------------------- | -------- |
| 1    | 标题                  | 组ID、成员、1行贡献说明                      | 0:20     |
| 2    | 动机与挑战            | 为何重要+2-3个关键挑战                       | 0:45     |
| 3    | 任务定义（输入/输出） | 输入/输出+系统边界（轮次、自由输入范围）     | 0:45     |
| 4    | 数据                  | 来源、规模、预处理、模式、训练/验证/测试划分 | 0:45     |
| 5    | 方法概述              | 架构：NLU→状态→NLG→一致性→输出               | 1:00     |
| 6-8  | 关键模块              | 意图/生成控制/一致性（每页1个）              | 2:15     |
| 9    | 实时演示              | 脚本化的2-3轮交互+显示状态更新               | 2:30     |
| 10   | 评估与总结            | 指标+1个失败案例+下一步+总结                 | 1:00     |

### **最终报告：推荐整体结构（在正文限制内）**

**建议章节大纲（可用作标题）**

1. **引言**：问题、任务输入/输出、贡献（3个要点）
2. **相关工作/背景**（可选）：只写您使用的内容
3. **任务与数据**：来源、规模、模式、预处理、划分
4. **方法**：NLU、状态、NLG、一致性（每个模块明确定义输入/输出）
5. **系统实现**：技术栈、推理流程、界面、工程细节
6. **实验与评估**：设置、指标、对比/消融、案例研究
7. **讨论与局限**：失败模式和未来改进
8. **结论**：总结关键发现
9. **贡献**：成员角色（姓名+学号+具体工作）

---

### **如何撰写方法部分（学术+可复现）**

**为每个模块使用一致的模板**
- **输入/输出**：精确定义输入和输出内容
- **方法**：模型/规则/提示设计（基线+改进）
- **理由**：此设计为何能解决您的挑战
- **失败模式**：哪些输入仍会破坏它

**您可以包含的具体"证据"**
- 状态模式+更新规则（附一个示例）
- 一致性检查清单（5-10条规则或检查）
- 提示模板/约束格式（附录或图表）
- 日志：每轮意图、状态、延迟、修复标记

*示例措辞：*  
我们将玩家输入理解形式化为对K种动作类型的意图分类。给定预测的意图和叙事状态，我们更新结构化记忆并在显式一致性约束下生成下一段剧情。

---

### **实验与评估部分应包含什么（最重要的部分）**

**最低信息集（没有它，严谨性难以证明）**
- **实验设置**：数据划分、轮次数、环境/硬件、模型配置
- **指标定义**：如何量化一致性/连贯性、延迟协议、意图准确性
- **结果表**：至少一个（包含单位和样本量）
- **案例研究**：一个成功+一个失败（失败+分析非常学术）

**对比/消融（可选但高价值）**
- 无一致性 vs 有一致性
- 提示设计变体
- 解码策略（温度/top-p/束搜索+重排）

**展示提示**
- 保持表格简洁易读
- 使用一个架构图+一个结果表+一个案例图
- 使用一致的案例格式：输入/状态/输出/问题/修复
